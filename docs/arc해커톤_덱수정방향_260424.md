## 1. 🔴 우선순위 1 — 슬라이드 6 가격 공식 (overclaim)

### 1-1. 문제

**현재 (덱 line 468, 532)**:
```html
line 468: <div class="li">...<b>Variable pricing</b> — confidence × notional × take-rate</div>
line 532: <div class="formula">price = <b>confidence</b> × <b>notional</b> × <b>|premium|</b> × <b>take_rate</b>
          &nbsp;&nbsp;→&nbsp;&nbsp;clipped to [$0.0005, $0.010]</div>
```

**실제 코드 (`consumers/executor_agent/pricing_policy.py:217,233`)**:
```python
base = FEE_ROUND_TRIP * notional_usd
price = max(PRICE_FLOOR, min(PRICE_CAP, base * ACTION_MULTIPLIERS[action_idx]))
# ACTION_MULTIPLIERS = [0.75, 1.0, 1.5, 2.5]
```

즉 **`confidence`는 가격 공식에 들어가지 않음**. UCB1 bandit가 premium-rate 상태를 보고 multiplier를 고를 뿐. Karpathy/Vitalik 렌즈에서 이건 심사관이 20초만에 발견할 수 있는 과장.

### 1-2. 수정 방향

**line 468 변경**:
```html
<div class="li">
  <span style="width:6px;height:6px;border-radius:50%;background:var(--accent-cyan)"></span>
  <b>Variable pricing</b> — UCB1 bandit picks Q-multiplier from {0.75, 1.0, 1.5, 2.5}
</div>
```

**line 532 변경**:
```html
<div class="formula">
  price = <b>fee_base</b> × <b>Q<sub>mult</sub></b>
  &nbsp;&nbsp;→&nbsp;&nbsp;clipped to [$0.0005, $0.010]
</div>
<div class="formula-sub" style="font-size:13px;color:var(--muted);margin-top:4px;">
  fee_base = FEE_ROUND_TRIP × notional_usd &nbsp;·&nbsp;
  Q<sub>mult</sub> ∈ {0.75, 1.0, 1.5, 2.5} via UCB1 over premium-rate state
</div>
```

**근거**: SUBMISSION.md §7 을 같은 방향으로 이미 정직화 완료. 덱과 문서 동기화해야 심사관이 두 소스 대조해도 일관됨.

### 1-3. 슬라이드 6 `qbar-label` 2개 (line 517, 527) — 유지 가능

"Low-confidence signal" / "High-confidence, high-notional" 라벨 자체는 **시각화 용 상태 구분**이지 공식 주장이 아니므로 그대로 둬도 됨. 단 심사관이 대조할 수 있으므로:
- `Low-confidence signal` → `Low premium-rate state`
- `High-confidence, high-notional` → `High premium-rate state`

로 바꾸면 **UCB1 state가 premium-rate buckets임**을 시각적으로도 정합.

## 2. 🟡 우선순위 2 — CSS orphan block (line 155-160)

### 2-1. 문제

`.margin-row` 정의가 `}` 하나 더 있고 (line 156이 닫혀서 끝난 뒤), line 157-160 이 **selector 없이 떠 있는 property 블록**이다:

```css
    .margin-row {
      display: grid; grid-template-columns: 240px 1fr 170px 36px;
      gap: 18px; align-items: center;
      padding: 9px 0;
      font-size: 17px;
      border-top: 1px solid var(--border);
    }                                          ← line 156 닫힘
      font-family: var(--font-mono); font-size: 19px;    ← line 157 orphan
      padding: 10px 0;                                    ← line 158 orphan
      border-top: 1px solid var(--border);                ← line 159 orphan
    }                                                     ← line 160 extra close
    .margin-row:first-of-type { border-top: none; }
```

### 2-2. 가능한 원인 (Claude Design 판단)

두 가지 해석 가능:
- **A. 원래 `.margin-row.total` 같은 modifier 셀렉터가 누락**된 것. `font-family: mono; font-size: 19px;` 는 합계 행을 mono로 키우려던 의도 같음.
- **B. 실수로 중복 selector 블록이 병합됨**. 이 경우 line 157-160 전체 삭제가 정답.

### 2-3. 권장 수정

**Option A (의도 보존 — 권장)**: 셀렉터 추가
```css
    .margin-row.total {
      font-family: var(--font-mono); font-size: 19px;
      padding: 10px 0;
      border-top: 1px solid var(--border);
    }
```

HTML에서 합계 행이 있는지 확인해야 함. `<div class="margin-row ... total">` 형태로 쓰이고 있으면 A, 아니면 B.

**Option B (안전 — 기능 확인 어려우면)**: line 157-160 전체 삭제
```css
    .margin-row {
      /* ... 기존 규칙 유지 ... */
      border-top: 1px solid var(--border);
    }
    .margin-row:first-of-type { border-top: none; }
```

브라우저에 렌더는 되지만 dev-tool / css-lint 에서 warning. 제출물에서 보기 안 좋음.

### 2-4. 검증

`npx stylelint AlphaLoop_Deck.html` 또는 Chrome DevTools → Computed panel 에서 `.margin-row.total` 규칙이 실제 어떤 element에 적용되는지 확인.
